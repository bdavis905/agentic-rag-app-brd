# Module 4: Metadata Extraction (Dynamic Schema)

## Complexity: ⚠️ Medium

**Goal:** LLM extracts structured metadata during ingestion based on a configurable schema stored in the database. Metadata is stored on documents and propagated to chunks. Retrieval supports metadata filtering. Frontend shows metadata in an expandable detail panel.

**Key decisions:**
- Schema is DB-stored JSON in `global_settings` (editable via API, no dedicated UI)
- Ships with sensible defaults (title, summary, document_type, topics, language)
- Supported field types: `string`, `list`, `enum`, `number`, `boolean`
- Same LLM as chat for extraction
- No backfill for existing documents

## Task 1: Database Migration

**Create:** `supabase/migrations/20260125000000_metadata_extraction.sql`

```sql
-- 1. Add metadata column to documents
ALTER TABLE documents ADD COLUMN metadata JSONB DEFAULT '{}';

-- 2. Add metadata_schema to global_settings with defaults
ALTER TABLE global_settings ADD COLUMN metadata_schema JSONB DEFAULT '[
  {"name": "title", "type": "string", "required": true, "description": "A concise, descriptive title for the document"},
  {"name": "summary", "type": "string", "required": true, "description": "A 1-3 sentence summary of the document content"},
  {"name": "document_type", "type": "enum", "required": true, "description": "The category of this document", "enum_values": ["article", "tutorial", "reference", "notes", "report", "essay", "code", "other"]},
  {"name": "topics", "type": "list", "required": true, "description": "3-7 key topics or themes covered in the document"},
  {"name": "language", "type": "string", "required": false, "description": "ISO 639-1 language code of the document (e.g. en, es, fr)"}
]';

-- 3. Update match_chunks with optional metadata filter
CREATE OR REPLACE FUNCTION match_chunks(
    query_embedding vector,
    match_threshold float,
    match_count int,
    p_user_id uuid,
    p_metadata_filter jsonb DEFAULT NULL
) RETURNS TABLE (
    id uuid, document_id uuid, content text,
    chunk_index int, metadata jsonb, similarity float
) LANGUAGE plpgsql AS $$
BEGIN
    RETURN QUERY
    SELECT c.id, c.document_id, c.content, c.chunk_index, c.metadata,
           1 - (c.embedding <=> query_embedding) AS similarity
    FROM chunks c
    WHERE c.user_id = p_user_id
      AND 1 - (c.embedding <=> query_embedding) > match_threshold
      AND (p_metadata_filter IS NULL OR c.metadata @> p_metadata_filter)
    ORDER BY c.embedding <=> query_embedding
    LIMIT match_count;
END;
$$;

-- 4. GIN index for fast JSONB containment queries on chunks
CREATE INDEX idx_chunks_metadata ON chunks USING gin (metadata jsonb_path_ops);
```

**Validation:** `supabase db push` succeeds. Existing match_chunks calls (4 params) still work. `metadata_schema` column has default value.

## Task 2: Schema Definition Models

**Create:** `backend/app/models/metadata.py`

Pydantic models for the schema definition itself (not the extracted metadata):

```python
class MetadataFieldDefinition(BaseModel):
    name: str                              # Field key
    type: Literal["string", "list", "enum", "number", "boolean"]
    required: bool = True
    description: str                       # Used in LLM prompt
    enum_values: list[str] | None = None   # Only for type="enum"

# Type alias for the full schema
MetadataSchema = list[MetadataFieldDefinition]
```

Also include:
- `get_metadata_schema(supabase) -> MetadataSchema` — loads schema from global_settings
- `validate_metadata(data: dict, schema: MetadataSchema) -> dict` — validates extracted metadata against the schema, applies defaults for missing non-required fields

## Task 3: Metadata Extraction Service

**Create:** `backend/app/services/metadata_service.py`

- `extract_metadata(text: str, user_id: str) -> dict`
- Loads the metadata schema from `global_settings`
- Builds a dynamic extraction prompt from the schema fields (name, type, description, constraints)
- Uses first 8000 characters of document text
- Calls configured LLM with `response_format={"type": "json_object"}`, `temperature=0.0`
- Validates response against the schema using `validate_metadata()`
- Falls back to defaults on failure (required strings → "Unknown", required lists → [], etc.)

**Dynamic prompt construction example:**
```
Extract the following metadata fields from the document:

1. title (string, required): A concise, descriptive title for the document
2. document_type (enum: article|tutorial|reference|..., required): The category of this document
3. topics (list of strings, required): 3-7 key topics or themes
...

Return a JSON object with these exact keys.
```
