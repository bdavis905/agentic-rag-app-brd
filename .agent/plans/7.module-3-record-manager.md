# Module 3: Record Manager

**Complexity:** ✅ **Simple** - Clean logic, no complex diffing

## Goal

Prevent duplicate ingestion and handle re-uploads cleanly. When a user re-uploads a file:
- **Identical content** → skip processing entirely (return existing document)
- **Modified content** → delete all old chunks, re-chunk and re-embed from scratch

The "delete all + re-embed" approach for modified files is simpler and avoids orphan chunks entirely — no risk of stale vectors lingering from a previous version.

## Current Problem

Every upload creates a brand new document + chunks, even if the file was already uploaded. Uploading `notes.txt` three times produces 3 document records and 3x the chunks — all appearing in retrieval results and degrading quality.

## Tasks

### Task 1: Database Migration — Add Content Hash + Unique Constraint

**What:** Add `content_hash` (SHA-256) to the `documents` table. Add a unique constraint on `(user_id, filename)` to enforce one-document-per-filename-per-user.

**Migration file:** `supabase/migrations/20260124000000_record_manager.sql`

```sql
-- Content hash on documents (SHA-256 of full file content)
ALTER TABLE documents ADD COLUMN content_hash TEXT;

-- Index for fast hash lookups
CREATE INDEX idx_documents_content_hash ON documents(user_id, content_hash);

-- Unique constraint: one document per filename per user
-- Re-uploads update the existing record rather than creating duplicates
ALTER TABLE documents ADD CONSTRAINT uq_user_filename UNIQUE (user_id, filename);
```

**Validation:**
- Run `supabase db push` — migration applies without errors
- Existing documents still queryable
- `content_hash` is nullable (backward-compatible with existing rows)

### Task 2: Record Manager Service

**What:** Create `backend/app/services/record_manager.py` with the core deduplication logic.

**Functions:**

1. `compute_file_hash(content: bytes) -> str` — SHA-256 hex digest of raw file bytes
2. `check_existing_document(user_id: str, filename: str) -> dict | None` — Look up existing doc by user + filename
3. `determine_action(existing_doc: dict | None, new_hash: str) -> Literal["new", "skip", "update"]` — Decide what to do:
   - No existing doc → `"new"`
   - Same content_hash → `"skip"`
   - Different content_hash → `"update"`
4. `delete_existing_chunks(document_id: str) -> int` — Delete all chunks for a document, return count deleted

**Validation:**
- `compute_file_hash` returns consistent hex string for same input
- `determine_action` returns correct action for each scenario
- `delete_existing_chunks` removes all chunks for a given document

### Task 3: Update Upload Endpoint with Deduplication

**What:** Modify `POST /documents/upload` to check for duplicates before creating a new document.

**Updated flow:**
1. Read file content, validate as before
2. Compute `content_hash = compute_file_hash(content)`
3. Check for existing document with same `(user_id, filename)`
4. **If exists and same hash** → return existing document with `"action": "skipped"` (no processing)
5. **If exists and different hash** →
   - Delete all existing chunks for that document
   - Re-upload file to storage (overwrite)
   - Update document record: new `content_hash`, `file_size`, reset `status` to `"pending"`, clear `error_message`
   - Trigger background processing
   - Return document with `"action": "updated"`
6. **If new** → create document record with `content_hash`, proceed as before, return with `"action": "created"`

**Response additions:**
- Add `action` field to response: `"created"`, `"skipped"`, or `"updated"`

**Validation:**
- Upload `test.txt` → response has `action: "created"`, document processed normally
- Upload `test.txt` again (same content) → response has `action: "skipped"`, no reprocessing
- Upload `test.txt` with different content → response has `action: "updated"`, old chunks deleted, new chunks created

### Task 4: Update Ingestion Service — Store Content Hash

**What:** Minor update to `process_document` to store `content_hash` on the document record after processing.

**Changes:**
- The content hash is already set by the upload endpoint, so the ingestion service just needs to ensure it's preserved when updating status to `"completed"`
- No chunk-level hashing needed — chunks are always fresh on each processing run

**Validation:**
- After processing completes, document record has `content_hash` populated
- Re-processing after update creates all-new chunks (no leftover old ones)

## Architecture Summary

```
Upload Request
     │
     ▼
[Compute file hash]
     │
     ▼
[Check existing doc by user_id + filename]
     │
     ├── No existing doc ──────────► "new" → create doc + full processing
     │
     ├── Same content_hash ────────► "skip" → return existing doc immediately
     │
     └── Different content_hash ───► "update" → delete ALL old chunks
                                                  │
                                                  ▼
                                         [Re-upload file to storage]
                                         [Update doc record]
                                         [Full re-processing]
                                                  │
                                                  ▼
                                         [Chunk → Embed → Store]
                                         (all fresh, no orphans)
```
