# Module 11: Ingestion Pipeline Performance Optimization

**Complexity:** ⚠️ Medium - Safe optimizations to existing code with clear performance benefits. No schema changes required.

## Overview

Optimize the document ingestion pipeline for better throughput and resource efficiency. These changes target four key areas: concurrency control, batch sizing, memory efficiency, and parallelization of independent operations.

**Expected Improvements:**
- 5x reduction in embedding API round trips (batch size increase)
- ~50% reduction in peak memory usage for large files (eliminate double loading)
- Better throughput under concurrent uploads (semaphore prevents overload)
- Faster processing time per document (parallel metadata + chunking)

**Risk Assessment:** Low - All optimizations are additive or improve existing patterns without changing external APIs or database schemas.

## Task 1: Add Concurrency Limits with Semaphore

**Problem:** When multiple users upload large documents simultaneously, the system can exhaust memory by processing all documents concurrently without limits.

**Solution:** Add an `asyncio.Semaphore` to limit concurrent document processing to a configurable number (3-5 tasks).

**Modify:** `backend/app/services/ingestion_service.py`

### Changes:

1. Add module-level semaphore constant and instance:
```python

MAX_CONCURRENT_DOCUMENTS = 3
_processing_semaphore = asyncio.Semaphore(MAX_CONCURRENT_DOCUMENTS)
```

2. Wrap the `process_document` function body with semaphore acquisition:
```python
async def process_document(document_id: str, user_id: str, file_bytes: bytes | None = None) -> None:
    """
    Process an uploaded document: extract text, chunk, embed, and store.
    Uses semaphore to limit concurrent processing.
    """
    async with _processing_semaphore:
        # ... existing implementation ...
```

1. **Unit Test:** Upload 5+ documents simultaneously and verify only 3 are in "processing" status at any time
2. **Memory Test:** Monitor memory usage during concurrent uploads - should stay bounded
3. **Functional Test:** All documents eventually complete processing successfully

## Task 3: Eliminate Double File Loading

**Problem:** Files are read during upload (for validation/hashing), then downloaded again from Supabase storage in the background task. This doubles memory usage for large files.

**Current Flow:**
1. `documents.py`: Read file content into memory for validation
2. `documents.py`: Upload content to Supabase storage
3. `ingestion_service.py`: Download content from Supabase storage again

**Solution:** Pass file bytes directly to the background task, eliminating the redundant download.

### Changes:

**Modify:** `backend/app/routers/documents.py`

1. Update background task calls to pass file bytes:

```python
# Line 112 (update case)
background_tasks.add_task(process_document, existing_doc["id"], current_user.id, content)

# Line 150 (new document case)
background_tasks.add_task(process_document, document["id"], current_user.id, content)
```

**Modify:** `backend/app/services/ingestion_service.py`

1. Update function signature to accept optional file bytes:

```python
async def process_document(document_id: str, user_id: str, file_bytes: bytes | None = None) -> None:
```

2. Modify file loading logic to use passed bytes when available:

```python
# Replace lines 62-63:
# Download file from storage
storage_path = doc["storage_path"]
file_bytes = supabase.storage.from_("documents").download(storage_path)

# With:
# Use passed bytes or download from storage (fallback for re-processing)
if file_bytes is None:
    storage_path = doc["storage_path"]
    file_bytes = supabase.storage.from_("documents").download(storage_path)
```

### Validation:

1. **Memory Test:** Monitor peak memory during large file upload - should be ~50% lower
2. **Functional Test:** Upload documents of various sizes, verify all process correctly
3. **Fallback Test:** Manually trigger `process_document` without bytes parameter (simulating re-processing) - should still work by downloading from storage

```bash

# In a separate terminal, use Task Manager or:

  -H "Authorization: Bearer $TOKEN" \
```

## Task 4: Parallelize Metadata Extraction and Chunking

**Problem:** Metadata extraction (LLM call) and text chunking are currently sequential, but they are independent operations that can run in parallel.

**Current Flow (sequential):**
1. Extract text from document
2. Extract metadata via LLM (slow - network call)
3. Chunk the text

**Solution:** Run metadata extraction and chunking in parallel using `asyncio.gather()`.

**Modify:** `backend/app/services/ingestion_service.py`

### Changes:

Replace the sequential calls (lines 71-80):

```python
# Before (sequential)
# Extract metadata using LLM
doc_metadata = await extract_metadata(text, user_id)

# Update document record with extracted metadata
supabase.table("documents").update({
    "metadata": doc_metadata,
}).eq("id", document_id).execute()

# Chunk the text
chunks = chunk_text(text)
```

With parallel execution:

```python
# After (parallel)
# Run metadata extraction and chunking in parallel - they're independent
async def _extract_metadata_wrapper():
    return await extract_metadata(text, user_id)

def _chunk_wrapper():
    return chunk_text(text)

# Execute in parallel
loop = asyncio.get_event_loop()
doc_metadata, chunks = await asyncio.gather(
    _extract_metadata_wrapper(),
    loop.run_in_executor(None, _chunk_wrapper)
)

# Update document record with extracted metadata
supabase.table("documents").update({
    "metadata": doc_metadata,
}).eq("id", document_id).execute()
```

Note: `chunk_text` is synchronous, so we run it in an executor to avoid blocking the event loop while the metadata LLM call is in flight.

### Validation:

1. **Timing Test:** Measure processing time before and after - should see improvement roughly equal to the shorter of the two operations
2. **Functional Test:** Verify metadata and chunks are both correctly generated
3. **Error Handling:** Verify errors in either operation are properly caught and reported

```bash
# Time a document upload end-to-end
time curl -X POST http://localhost:8000/documents/upload \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@test-document.pdf"

# Wait for completion, then verify both metadata and chunks exist

curl http://localhost:8000/documents/{doc_id}/chunks -H "Authorization: Bearer $TOKEN" | jq 'length'
```

## Rollback Plan

If issues arise, each optimization can be independently reverted:

1. **Semaphore:** Remove `async with _processing_semaphore` wrapper
2. **Batch size:** Change `BATCH_SIZE` back to 50
3. **Double loading:** Remove bytes parameter, restore storage download
4. **Parallelization:** Replace `asyncio.gather` with sequential calls

All changes are backward-compatible and don't affect the database schema or API contracts.
